$ On-premies 와 Off-premies ->
On-premies 기업이 서버를 클라우드 같은 환경에서 운영하는 방식이 아닌, 자체적으로 보유한 서버에 직접 설치해 운영하는 방식을 말한다.
On-premies 시스템의 장점은 기업 비즈니스 정보를 보안성 높게 관리할 수 있다는 것이고, 단점은 시스템을 구축하는데 많은 시간과 노력이 들어간다.
On-premies 시스템의 단점은 사용자의 데이터를 직접 관리하기에 보안에 장점이 있다고 생각할 수 있지만, 본인이 직접 관리 하는 것 보다 보안성이 더 좋은 기업에 위탁 관리하는 것이 더 좋을 수 있다.
또한 직접 관리하기에 서버가 커질 수 록 유지보수하기 힘들어 진다.
Off-premies 는 On-premies의 반대말로 인터넷 네트워크에 연결된 서버팜이나 클라우드에서 원격으로 실행하는 환경을 말한다.
Off-premies 의 장점은 초기 비용이 크게 들어가지 않는다. 대부분 클라우드 서버를 관리하는 기업에 사용하는 만큼간 비용을 지불한다.
보안 서비스를 지원 받으며, 네트워크 보호, 위협 모니터링, 데이터 암호화등 클라우드 기업해서 대행 해준다.
Off-premies 의 단점은 장기적으로 보았을 때 클라우드 서버 유지비용이 서버 구축비용보다 더 많이 들 수 있다.
두번째로 클라우드를 통해 서버를 관리하는 회사는 본인이 소유한 서버가 없고 클라우드를 통해 작업해야히 때문에 유연성이 낮을 수 있다.
최근 동향은 On-premies 와 Off-premies 하나만 이용한다기 보단 두개를 같이 이용하는 방법도 많이 쓰인다.
보안이 중요한 서비스, 데이터는 On-premies 서버에 구성하고 그 외 서비스 이용환경은 Off-Premise에 저장함.

$ IDC ->
IDC는 Internet Data Center 의 약자로 인터넷 서비스를 위한 서버 관리 센터입니다.
IDC서버의 장점은 관리하기 편하다. 자체 서버를 설치해 사용할 경우 별도의 서버 관리 인력이 필요하지만. IDC 서비스를 이용하면 별도 인력이 크게 필요 없다.
두번째 물리적 대응에 강하다. 정전이나 지진, 인터넷 먹통등으로 서버 연결이 끊어지는 경우 가동이 어렵다. IDC 데이터 센터의 경우 이를 대비해 별도의 전원 시스템이나, 다양한 경로로 인터넷 연결을 해둔다.
IDC 서버와 클라우드 서버의 차이점은 클라우드 서버의 경우 가상 서버를 임대하는 방식을 사용하기에 서버 확장, 축소에 유연하다. 그러나 클라우드 서버의 경우 트레픽 사용량 만큼 요금을 적용하기에 트레픽이 많으면 서버 비용이 증가한다.
IDC서버는 물리적 서버를 단독으로 임대, or 판매하여 서버 관리, 인프라, 인력을 외주 형태로 제공하는 방식이다.
IDC서버는 자체 서버를 이용하기에 인터넷 비용과 서버 비용을 정액제로 부담한다.

$ IaaS ->
IaaS(서비스형 인프라)는 중앙에서 관리되는 클라우드, 스토리지, 컴퓨팅으 네트워킹 리소스를 필요에 따라 가상 환경에서 신속하게 사용 할 수 있도록 지원한다.
IaaS는 기존 On-premise 서비스 대신 퍼블릭 클라우드, 프라이빗 클라우드에서 호스팅 되고 필요에 따라 클라이언트에게 제공된다. 고객은 브라우저 기반 관리 콘솔과 API를 이용해 필요할 때 서버에 접속할 할 수 있다.
IaaS는 가상화 기술로 아주 빠른 시간 이내에 리소스를 확장, 축소가능해 유연하고 탄력적인 운영이 가능하다.
IaaS는 사용자가 많은 부분을 컨트롤하고 관리 할 수 있어 다양한 목적에 따라 인프라를 자유롭게 활용할 수 있지만, 인프라 운영에 대한 경험, 지식, 인력들이 어느정도 필요할 수 있다.

$ PaaS ->
PaaS는 Platform as a Service(플랫폼 기반 서비스)의 약자로 클라우드 컴퓨팅 서비스의 한 종류이다.
어플리케이이션 구측, 실행 및 관리하는 데 필요한 플랫폼을 제공하는 서비스이다. 개발자가 어플리케이션을 제작하고 실행 환경, 데이터베이스, 웹서버 등 기술적 측면을 고려하지 않고 어플리케이션을 배포할 수 있게 해준다.
PaaS는 개발도구, 미들웨어, 운영 체제, 데이터베이스 관리, 인프라가 포함된다. 제공자에 따라 다른 서비스들도 포함 될 수 있지만 이 5가지가 핵심 서비스이다.
개발도구는 소스 편집기, 디버거, 컴파일러등 필수 프로그램을 포함해 소프트웨어 개발에 필요한 툴을 제공한다. PaaS서비스에는 개발에 필요한 모든 서비스가 포함된다.
미들웨어 PaaS는 기본적으로 미들웨어가 포함되어 있으므로 개발자가 환경을 직접 구축할 필요가 없다.
운영체제 PaaS는 개발자가 작업하고 어플리케이션을 실행하는 운영체제를 제공한다.
데이터베이스 PaaS 공급자는 데이터베이스를 관리하고 유지한다. 공급자는 데이터베이스 관리 시스템도 클라이언트에게 제공된다.
인프라 PaaS는 클라우드 컴퓨팅 서비스 모델에서 laaS에 포함된 모든 것이 PaaS에도 포함되어 있다.
PaaS를 사용하는 이유는 개발자가 별 다른 사전 작업 없이 개발을 할 수 있어 앱 개발 시간 단축에 용의하다.
PaaS의 단점은 PaaS 제공자를 선택하고 다른 PaaS 제공자로 이전할 때 수 많은 비용과 자원이 들어간다.
비용 에측이 어려울 수 있다. 초기 비용이 적을 수 있지만 갑작스러운 트래픽 증가가 생기면 비용이 크게 증가할 수 있다.

$ SaaS ->
SaaS는 Software as a Servie의 약자로 클라우드 기반 소프트웨어 제공 모델이다.
SaaS는 클라이언트가 소프트웨어를 자신의 서버에 설치하고 유지보수 하지 않고 인터넷을 통해 어플리케이션에 접속 할 수 있다. 서버 구성, 보안, 업데이트 같은 기술 적 부분은 제공자가 책임진다. 클라이언트는 소프트웨어 사용 비용만 지불하면 된다.
SaaS의 장점은
클라우드 접근성이 좋다. 인터넷 연결이 되어 있는 모든 기기에서 SaaS에 접근할 수 있다.
초기 비용 투자가 적다. 사용자 수나 포함 기능에 따라 유연하게 요금을 책정하므로 기업에 필요한 만큼 비용을 지불하여 운영 비용 절감 가능하다.
확장성도 좋다. 인프라 업그레이드 비용이 들지 않고 별도 종량제 요금만 지불하면 큰 비용을 들이지 않고 더 많은 서버와 스토리지를 사용할 수 있음
SaaS 도입 시 고려할 사항은
네트워크 영향을 받는다 SaaS는 클라우드 서비스를 기바으로 하므로, 인터넷 연결이 끊어지면 서비스를 받을 수 있다.
인터넷 연결 속도에 영향을 받는다. 인터넷 연결 속도와 지연 시간에 영향을 받음.
기존 시스템이 있는 경우 처음부터 개발을 다시 시작해야한다.

$ IaaS, PaaS, SaaS의 차이점 ->
Iaas -> 클라이언트는 어플리케이션, 데이터, 미들웨어, 운영체제를 관리하고, 서비스 제공 업자는 가상화, 서버, 스토리지, 네트워킹을 관리한다.
PaaS -> 클라이언트는 어플리케이션과 데이터만 관리한다. 서비스 제공 업자는 그 밖 모든 것을 관리한다.
SaaS -> 서비스 업체가 모든 것을 관리해 간편하게 사용 가능함

$ 하이퍼바이저 ->
하이퍼바이저란 하나의 호스트 컴퓨터상 다수의 운영체제를 구동시킬 수 있는 하드웨어와 운영체제 사이 소프트웨어 가상화 플랫폼을 말한다.
하이퍼 바이저는 컴퓨터의 프로세서와 메모리, 등등 공유 자줜에 접근하는 방법에 대해 통제하는 길술을 가졌다.
하이퍼바이져의 역핳은 높은 수준의 관리 및 모니터링 도구에 대한 인터페이스를 제공하고 가상 운영체제간 간섭을 통제한다.
하이퍼 바이져의 구동 방식은 2개가 있다.
Tpye 1. 네이티브 또는 버에 메탈 하이퍼바이저라고 불린다. 호스트의 하드웨어에서 직접 구동되어 게스트 운영체제를 관리한다. 호스트의 하드웨어에서 직접 구해 게스트 운영체제를 관리 한다. 호스트 운영 체제 대신 가상머신 리소스는 하이퍼바이저에 의해 하드웨어에 사용된다.
Type 2. 기존 운영체제 환경에서 실행되는 소프트웨어 프로그램이다. 하이퍼바이저가 물리 컴터상의 호스트 운영체제 위에서 동작한다. 이 구조의 장점은 게스트 운영체제 종류에 대해 제약 사항이 없다. 다양한 운영체제를 게스트가 사용할 수 있다.

& rm -r, rm -rf ->
리눅스에서 rm 명렁어는 파일을 지울 때 사용하는 명령어다. rm -r은 리눅스에서 폴더 + 하위 파일 까지 지우는 명령어다.
rm -rf는 파일 폴더를 강제로 지워 준다.
rm -rf {경로 or 파일명} 을 입력하며 현재 디렉토리 내에 있는 파일을 모두 지워 버린다. 사용자에게 삭제 여부를 알려주지 않고, 심지어 읽기 전용으로 표시도니 쓰기 방지 파일도 삭제를 한다.
대부분 (rm -rf ./ ),(rm -rf /), (rm -rf . ) 점 하나 차이로 큰 사고가 발생한다. 두번째 /의 경우 현제 디렉토리까지 삭제 해버리는 명령어로 지우고 싶은 파일이 아닌 다른 파일까지 삭제할 수 있다. 삭제 시 복구하는 절차가 엄청 까다롭기게 필히 사용하는 상황이 아니라면 사응을 자제한다.

$ grep ->
grep 는 리눅스에서 특정 파일에서 지정한 문자열이나 정규표현식을 포함한 행을 출력해주는 명령어이다.
사용 예
$grep ‘찾을 내용’ “파일명” // 파일명에서 ‘찾을 내용 ‘ 문자열 찾기
$grep ‘찾을 내용’ “파일명1” “파일명2” // 파일명1, 파일명 2 에서 문자 찾기
$grep ‘찾을 내용’ * // 현제 디렉토리내에 잇는 모든 파일에서 문자 찾기
$grep ‘찾을 내용’ *.확장자명 // 디렉토리 내 확장자명을 가진 파일에서 문자 찾기

$ 파일 사용자와 그룹 ->
리눅스에서는 여러 사요자가 하나의 OS를 사용하는다중 사용자 시스템(multi -User_System)을 구현한다.
주로 서버로 사용 되는 리눅스시스템에서는 여러사용자가 하나의 OS에 접근하게 된다. 따라서 리눅스에서 사요자를 관리하고 사용자 그룹을 관리 하는 것이 중요하다.
리눅스는 모든 권한을 가진 super user가 있다. 일반적으로 root이름의 유저가 super user 이다.
어쨋거나 파일은 파일이기 때문에 파일에 권한을 부여해서 다른 유저가 파일에 접근 못하도록 할 수 있다.
ls -l 을 하면 디렉토리 내 파일과 파일 허가권이 나온다. 파일허가권은 rwx(소유자) rwx(그룹) rwx(그외 사용자로 ) 콘솔에 표시가 된다. //여기서 rwx는 R = 읽기, W = 쓰기, X = 실행이다.
접근 허가권은 chmod 명령어로 바꿀 수 있다.

$ sudoer 파일
sudoer 파일은 sudo 명령어를 사용할 수 있는 계쩡을 관리하는 설정 파일이다.
예를 들어 ls -l 을 해서 - r - - r - - - - - 이란 파일 이 있으면, 이 파일은 root 계정에 모든 권한이 걸려 있고, set uid 나 set gid가 걸려 있지 않은 파일이라, 일반 사용자가 이 파일을 읽는 것은 완전 불가능하다. 이 파일의 내용을 알아야 할 때 sudo su 명령어를 이용해 super user모드로 전환을 한 뒤 파일 을 열거나 chomd 명령어로 파일 권한을 변경 해야한다.

$ netstat ->
netstat 명령어란 전송 제어 프로토콜, 라우팅 테이블 등 수 많은 네트워크 인터페이스 또눈 소프트웨어 정의 네트워크 인터페이스, 네트어크 프로토콜 통계를 위한 네트워크 연결을 보여주는 명령 줄 도구다.
netstat 명령어의 옵션은
-a 현재 다른 서버 또는 pc와 연결, 리스닝 되어 있는 모든 포드 번호를 확인한다.
-n 현재 다른 서버 또는 pc와 연결 되어 있는 포트 번호를 확인한다.
-s ip, icmp, udp 프로토콜의 상태를 확인한다.
-u udp 프로토콜을 보여준다,
-t TCP 프로토콜을 보여준다.
-c 프로토콜에서 사용 PID를 보여준다.
-e 랜카드에서 송수신한 패킷의 정보를 확인한다.
-r 라우팅 테이블 확인과 연결되어 있는 포드 번호를 확인함

$ tmux -> 터미널 멀티플럭서의 줄임말로 터미널에서 개발을 진행할 때 유용한 기능을 제공하는 프로그래이다. 어떠한 터미널에서 session을 싱행 하면 그 세선 안에 작업창을 여러 개 나누어 병렬적으로 일을 수행할 수있다. 이렇게 터미널을 나누어 작업 하는 것이 tmux의 핵심 키워드라거 할 수 있다.
예로 서버에 시간이 오래 걸리는 작업을 백그라운드로 켜놓고 터미널을 종료 후 다른 작업을 해야할 때 자주 쓰인다.
